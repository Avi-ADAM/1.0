## 1. הקונסיירז' מתפצל לשניים — וזו האבחנה המרכזית

קונסיירז' אחד = **שני מנגנונים שלובים** שנהפכים לעצמאיים ברגע שמודלים אותם נכון:

```
                  ┌─────────────────────────────────────┐
                  │   החצי הציבורי / הגלובלי            │
                  │  (גילוי, AI, התאמה, פרובידרים)      │
                  │                                     │
   משתמש כותב    │  • חילוץ AI (Gemini → spec מובנה)   │
   "אני רוצה X" ─▶│  • התאמה ל-Pinecone (5 namespaces)  │
                  │  • התאמה לפרובידרים זרים           │
                  │  • feed פומבי של משאלות פתוחות     │
                  │  • Provider profiles                │
                  └──────────────┬──────────────────────┘
                                 │  הצעות חתומות נכנסות
                                 ▼
                  ┌─────────────────────────────────────┐
                  │   החצי הקבוצתי / הפרטי              │
                  │       === זה בדיוק Space ===        │
                  │                                     │
                  │  • חברי ה-Ratson (ratson-share)     │
                  │  • chat_forum (פורום-קבוצה)         │
                  │  • הצבעה פר-פריט (Vote events)      │
                  │  • Willingness entries              │
                  │  • Final breakdown (projection!)    │
                  │  • Tosplit + Haluka → תשלום         │
                  └─────────────────────────────────────┘
```

- **החצי הקבוצתי**: כל אירוע שמתרחש אחרי שמשתמש הצטרף ל-Ratson הוא עניין
  פנימי של הקבוצה. **זה בדיוק Space**. עוברים אותו ל-Phase 0+P2P בלי ויכוח.
- **החצי הציבורי**: גילוי, פרסום, AI, פרובידרים. זו הקטגוריה הקשה
  (§7 בתכנית הכללית). דורש טיפול נפרד — וגם רעיונות מחוץ לקופסה.

**ההשלכה המעשית**: ברגע שמשתמש הצטרף לקבוצה, הקונסיירז' הוא serverless-ready
מיד. הקושי כולו לפני ההצטרפות — בשלב שבו מישהו מנסה למצוא משהו ומישהו אחר
מנסה להציע אותו.

## 2. מיפוי 1-ל-1: שדה ב-Strapi → אירוע חתום

מבחינה סכמטית, כל המודל של `PLAN_SHARED_PURCHASE` מתורגם לאירועי DAG בלי
לאבד שום סמנטיקה. להפך — חלק מהדברים נעשים נקיים יותר.

| ישות/שדה ב-Strapi | אירוע (`subject.type` / `action`) | הערה |
|---|---|---|
| `Ratson` חדש | `ratson.create` | predicate: `joinKind, minJoiners, joinDeadline, consensusRule, partialConsensusFallback, willingnessModel, access_mode` |
| `Ratson.allowJoin/share_status` | חישוב מתוך לוג אירועים | סטטוס recruiting/locked = projection של אירועי `lock`/`expire` |
| `ratson-share` (per joiner) | `ratson.join` / `ratson.leave` | יוצר/מסיר חברות-Space — לא רק רשומה, אלא מתן מפתח ה-epoch |
| `Ratson.users_permissions_users` (mirror) | חברי ה-Space | אין יותר mirror — חברות נגזרת מאירועים |
| `chat_forum` הצמוד | `chat.message` events | אותו DAG, payload בלבד מוצפן |
| `ratson-proposal` | `proposal.create` (נחתם על-ידי המציע) | המציע יכול להיות חבר, פרובידר, או AI agent (§4) |
| `ratson_willingness_entry` (component) | `willingness.record` | חתום על-ידי המשתתף; שעות במקום כסף נשמרים יחד |
| `Vote` על proposal/פריט | `proposal.vote` עם `predicate.item_kind/idx` | בדיוק `tosplit.vote` של Phase 0, רק יעד שונה |
| `final_breakdown` (JSON ב-DB) | **projection טהורה** | לא נשמר! מחושב מאירועים על-פי `willingnessModel` |
| `confirmRatsonExecution` | `proposal.execute` (חתום על-ידי קוורום) | קוורום = כל המסכימים חתמו על אותו snapshot של ה-breakdown |
| `Sheirutpend`, `Tosplit`, `Haluka` | אותם reducers שכבר תוכננו ב-Phase 0 | רק יותר מהם |

**מה השתפר?** ה-`final_breakdown` היום הוא JSON ב-Strapi — לקוח לא יודע אם
החישוב נעשה נכון. במודל החדש זה projection דטרמיניסטי שכל לקוח מריץ
מקומית. אם פרובידר טוען "שילמתם פחות ממה שהסכמתם", אפשר להוכיח את זה
מתמטית — ה-`willingnessModel` (proportional_cap, pareto_sum, vickrey_light)
הופך לפונקציה ציבורית עם hash מפורסם. **תכונה שלא קיימת היום.**

## 3. הצד הציבורי: ארבע תת-בעיות שונות

הצד הציבורי לא מונוליטי. הוא ארבע בעיות נפרדות ולכל אחת פתרון אחר.

### 3.1 חילוץ AI — בעיית האורקל

`POST /api/concierge-extract` קורא ל-Gemini עם API key, מחזיר spec מובנה.
זה הדבר הכי "שרת" בכל הקודבייס.

**ארבע אפשרויות, לפי סדר אגרסיביות:**

| גישה | תיאור | מתי לבחור |
|---|---|---|
| **א. השרת כשירות** | משאירים את `concierge-extract` כשירות חיצוני שאפשר להחליף. ה-AI key לא יכול לחיות בלקוח. | ברירת מחדל ל-S2–S4. עובד עכשיו, לא דורש כלום |
| **ב. LLM בלקוח** | Gemini Nano (Chrome built-in AI, יציאה הדרגתית), Phi-3-mini/Qwen ב-ONNX, או WebLLM. מספיק לחילוצים פשוטים | כש-Chrome built-in AI יהפוך לזמין-ברירת-מחדל. שדרוג שקוף — ה-API נשאר אותו |
| **ג. רשת אורקלים חתומה** | מספר שירותי חילוץ (אחד שלנו, אחד של קהילה, אחד open-source). כל אחד מחזיר extraction **חתום**. הלקוח אוסף k מתוך n, ממוצע/הצבעה. | S5. כשהמודל הופך לנכס קהילתי |
| **ד. מצב "מלווה"** (כבר מתועד ב-PLAN_HOMEPAGE_MASTER §7.1) | אדם/בוט קהילתי משלים ידנית. **חתום על-ידי המלווה** | זמין מיד; משדרג את "המלווה" ל-trust delegation מפורש (§4) |

**רעיון מחוץ לקופסה — extraction כ-claim, לא כעובדה**: בקונסיירז' של היום
ה-extraction נשמר ב-Ratson כאילו הוא ה"אמת". במודל החתום, extraction הוא
**טענה חתומה של אורקל**: "הסוכן X טוען שהמשאלה הזו מורכבת מ-Y". המשתמש
יכול לקבל, לדחות, או להחליף. ההיסטוריה נשמרת. זה הופך את ה-AI מ"קסם" ל-
"חבר אחד מתוך כמה שהמשתמש סומך עליו, בנושא ספציפי". זה גם פותר את בעיית
ה-hallucinations: הם הופכים גלויים וזמינים לתיקון.

### 3.2 גילוי וקטורי — Pinecone

`enrichWish` ו-`suggestRatsonsToUser` משתמשים ב-Pinecone (Gemini 3072d,
5 namespaces). זה האינדקס המרכזי של המערכת.

**המסלול:**

1. **S2**: Pinecone נשאר. הוא **read-only מבחינת המודל** — האירועים החתומים
   הם המקור, Pinecone הוא view שנגזר. אם Pinecone שיקר/נפל — הנתונים שלמים.
2. **S3**: הוספת מסלול-עוקף — **embeddings בלקוח** (`@xenova/transformers`
   או דומה, ~50MB מודל, רץ פעם אחת בלקוח). הלקוח שולח לאינדקס embedding,
   לא טקסט. מטא-דאטה (latency, מטא) נשארת אצל האינדקסר.
3. **S4**: **ריבוי אינדקסרים מתחרים** — כל קהילה יכולה להריץ אינדקסר משלה
   על ה-feed הפומבי. הלקוח מצליב k אינדקסרים. אינדקסר משקר נחשף (החתימה
   על המשאלה לא מסתדרת עם מה שהאינדקסר מחזיר).
4. **S5**: אינדקסים פתוחים על רשת relays (כמו מנועי חיפוש על Nostr היום).

**העיקרון**: בכל שלב, גילוי הוא תת-אופטימלי קצת אבל **שמיש**. אנחנו לא
מאבדים את האינדקס לפני שיש לנו תחליף שעובד.

### 3.3 פרובידרים שאינם בפלטפורמה — הגשר ל-web הפתוח

זה היחיד שלא ניתן לבזר באופן צרוף: פרובידרים שאינם משתמשים עדיין צריכים
לקבל הזמנה. **רוב הפרובידרים בתחילת חיי המוצר לא יהיו משתמשים.**

**המסלול:**

- **S2–S5**: שולח-ההזמנה (אימייל / SMS / וואטסאפ) הוא **השירות האחרון
  שנשאר שרת**. בדומה ל-Web Push (אמת לא נעימה #1 בתכנית הכללית), זו דרישת
  ארכיטקטורת ה-web, לא בחירה שלנו.
- ההזמנה מכילה קישור ייחודי לעמוד `/r/<wishId>` — `read-only` של המשאלה
  (decryption key מוטמע בקישור כ-fragment, כמו share-link של 1Password).
- תגובת הפרובידר היא **`proposal.create` חתום**. אם הפרובידר עדיין לא רשום,
  המערכת מייצרת לו זוג מפתחות מתפעלי בדפדפן ומציעה לו לאמץ את החשבון
  ("הצעת לדנה, רוצה לעקוב אחר ההצעה שלך? לחץ כאן").
- ב-S5: ה-feeds הפומביים על רשת relays פתוחה; פרובידר יכול לעקוב אחר
  feed ללא הזמנה אישית. ההזמנה נשארת רק כ-NUDGE.

### 3.4 Feed פומבי של משאלות פתוחות

`access_mode='public'` של Ratson. רוב המשאלות בקונסיירז' יהיו פומביות —
זו עיקר המטרה. **לכן אין כאן בעיית פרטיות אמיתית** — הפרסום הוא מעשה
פומבי מהגדרה. הפתרון פשוט יחסית: אירועי `ratson.publish` נכנסים ל-feed
פתוח. כל לקוח/אינדקסר יכול לעקוב. ב-S5 זה רץ על relays פתוחים.

**מה כן צריך זהירות**: רידקציה. גם משאלה פומבית לא בהכרח חושפת את כל
המידע (שם פרטי של המבקש, כתובת מדויקת). הפרסום הוא אירוע נפרד שכולל גרסה
**מצומצמת** של ה-Ratson — לא רק טוגל גלוי/נסתר.

## 4. רעיונות מחוץ לקופסה ייחודיים לקונסיירז'

הקונסיירז' פותח דלתות שמודל הצבעה רגיל לא פותח, ברגע שעוברים לחתימות.

### 4.1 משאלה כ-covenant (התחייבות-משותפת ניתנת-לאימות)

כשקבוצה "נועלת" Ratson, כל החברים חותמים על אותו snapshot של ה-Ratson
(`ratson.lock` במצב שבו N חברים חתמו חתימה שניה). זו **התחייבות
מולטילטרלית קריפטוגרפית**: "אנחנו N אנשים מתחייבים לקנות X אם פרובידר
מציע עד תאריך Y במחיר עד Z".

**למה זה משנה כלכלית**: היום פרובידר רואה "Ratson פתוח, יש 8 חברים".
אין לזה משמעות חוזית — לחיצה על "אני בפנים" היא לחיצה. במודל החתום, ה-8
חתימות הן הוכחה — דמוקיט עם ערך כלכלי. **זה Groupon הפוך וטוב יותר**:
ב-Groupon ההתחייבות באה אחרי שהמחיר מתבצע ("יחול אם 100 ייצרפו"). אצלנו
ההתחייבות באה לפני, כתעודת אמינות.

### 4.2 רשת אורקלי חילוץ — pluralism כ-feature

(הוזכר ב-§3.1, כאן בהרחבה.) חילוץ AI הוא נקודה בה ה"אמת" לא חד-משמעית.
פרשנויות שונות לאותו טקסט הן לגיטימיות. במקום אורקל יחיד, רשת:

- **סוכנים מתחרים**, כל אחד עם פרופיל ידוע (Gemini / Claude / מודל
  פתוח / מקומית). כל extraction חתום.
- **המלצות שלי**: סוכן שאני אישית "אהבתי את ההצעות שלו" עולה ל-top של רשימת
  הסוכנים שלי. הסכמה מפורשת = רכישת אמון, גם זו פעולה חתומה.
- **diff בין סוכנים** = פיצ'ר. "שני סוכנים זיהו 'טיול' כקטגוריה, אחד
  הוסיף 'הסעה'. רוצה להוסיף?"
- בעולם שבו AI הופך לסטנדרט, יכולת לבחור _איזה_ AI ולאמת את הפלט שלו
  היא יתרון תחרותי משמעותי.

### 4.3 הצעת פרובידר כ-`signed offer` — מוניטין קריפטוגרפי

הצעה (`ratson-proposal`) שמוגשת על-ידי פרובידר היא היום רשומה ב-DB.
במודל החדש זה אירוע חתום שמכיל הבטחה: "אספק X תמורת Y עד תאריך Z".

- **לאחר ביצוע**, הקבוצה חותמת `proposal.fulfilled` או `proposal.breached`.
- **המוניטין של הפרובידר = אוסף האירועים החתומים שלו**. ניתן לאימות מבחוץ,
  לא ניתן לזיוף, לא ניתן למחיקה על-ידי המפעיל (גם לא על-ידינו).
- שונה אסטרטגית מ-Yelp/Trustpilot: שם המוניטין שייך לפלטפורמה. כאן הוא
  שייך לפרובידר ועוקב אחריו לכל מקום.

זו תכונה שמושכת פרובידרים מקצועיים אל הפלטפורמה. **לא בגלל שהם רוצים
P2P** (לא אכפת להם) — אלא בגלל שהם מקבלים נכס בר-העברה (portable reputation).

### 4.4 Bid aggregation בין-קבוצות

פרובידר יכול לחתום הצעה שמותנית בכמה Ratsons שונים: "אם הקבוצה A מאשרת
ספסל וגם הקבוצה B מאשרת ספסל, אבוא לכפר שלכם וזה יעלה לכל אחת חצי".
פרובידר הופך לרכז קבוצות. **בלי שום פלטפורמה מרכזית שמתאמת.** כל הצדדים
רואים אותה הצעה חתומה.

זה מעבר ל-Groupon: ההצעה לא מותנית בכמה אנשים, אלא בכמה _קבוצות מאומתות_.
מציאות שלא קיימת היום בגלל מורכבות התיאום — אבל קלה כשיש לוג חתום.

### 4.5 Vickrey שאפשר לאמת

מודל `vickrey_light` ב-PLAN_SHARED_PURCHASE.md היה "מסובך להסבר" כי לא
ברור איך אדם בודק שהשרת חישב נכון. ב-projection דטרמיניסטי שכל לקוח רץ —
**Vickrey הופך לאמין באופן ויזואלי**: לחץ → ראה מי הציעו מה → ראה את
החישוב → השוואה. מודלי-תמחור אקזוטיים שאיש לא הצליח לעלות בעולם ה-SaaS
(כי "מי בודק שהשרת לא רימה?") נעשים אטרקטיביים.

זה אגב פותח דלת ל**מנגנונים כלכליים שיתופיים מתקדמים** — חישוב Pareto
optimal, קצובת Lindahl, voting-by-veto — שדורשים נראות מתמטית. דרך
לחדש מה ש-FreeMates מציעה ככלכלת-קואופ.

### 4.6 הזמנה הופכת חבר זמני

קישור `/r/<wishId>` שמופץ במייל לפרובידר יוצר **חבר-Space זמני**: מפתח-
מכשיר שמייצרים בדפדפן שלו, מקבל מפתח-epoch מצומצם (רק לקריאת ה-Ratson
ולכתיבת `proposal.create`). מאפשר אינטראקציה מלאה לפני אונבורדינג רשמי.
אם הוא רוצה לאמץ → המפתח מועלה למעמד "חבר רגיל" באירוע חתום.

זה משכפל את הזרימה ה"קונסיירז' מלווה" המתועדת בלי הצורך באדם מאחורי-
הקלעים. תשתית טכנית שמקבילה לעקרון המוצרי "כניסה רכה".

## 5. מיפוי לסולם ההדחה של השרת (S0–S5)

מסונכרן עם הטבלה של §1 ב-PLAN_serverless_p2p_data, אבל ספציפי לקונסיירז':

| שלב | מה משתנה בקונסיירז' | קושי |
|---|---|---|
| **S1** (Phase 1 הקיים) | אירועי הצבעה על proposals ועל פריטי-proposals חתומים במקביל ל-`addVote`. `evaluateRatsonConsensus` מתעד גם הסכמה חתומה לאודיט. | קטן — וריאציה על `addVote`-flow של tosplit |
| **S2a** | ה-Space של ה-Ratson נוצר עם חברים, chat_forum, willingness, proposal votes. השרת = mirror. ה-AI extraction עדיין שרתי. | בינוני — צריך להגדיר Space-genesis פר-Ratson חדש |
| **S2b** | אירוע `proposal.create` נחתם על-ידי AI-agent / פרובידר / מלווה. `final_breakdown` = projection, לא DB row. הסרת `Ratson.users_permissions_users` mirror. | בינוני — projection במקום DB column |
| **S3a** | ה-Ratson עובר ל-Space מוצפן. extraction נשאר שרתי אך מקבל את **המשאלה המוצפנת** ומחזיר extraction חתום בקריאה אחת (request מוצפן + key חד-פעמי) | מתון |
| **S3b** | מודל אורקל: client בוחר מאיזה שירות חילוץ לבקש. כתובת השירות מתועדת באירוע. תיעוד "מי חתם על איזה extraction" שקוף. | בינוני |
| **S4** | אינדקסרים מתחרים. embeddings בלקוח. שחזור חברתי לפרובידרים מאומתים. הזמנה ל-feed פתוח (Nostr-style) ולא רק email. | גבוה — דורש שינוי תפיסתי לפרובידרים |
| **S5** | רשת אורקלים פתוחה. AI נע בין שירותים. Pinecone מוחלף ברשת אינדקסים-קהילתיים. השרת היחיד שנשאר: web push + email bridge לפרובידרים-אופליין | מחקרי |

## 6. מה לעשות עכשיו — לפני המעבר

הקונסיירז' זה הפיצ'ר הדגל; אסור לשבור אותו. צעדים מעשיים שאפשר לעשות
**במקביל ל-Phase 1** של התכנית הקיימת, בלי לעכב את ההשקה:

1. **לחתום על Vote events של `voteOnRatsonProposal`** (shadow, כמו tosplit
   ב-Phase 1). אפס סיכון, ערך גובר אם פרובידר חולק על תוצאה.
2. **לאחסן `final_breakdown` כ-projection במקביל ל-JSON ב-DB**. כפילות
   זמנית; מאפשרת השוואה (השרת מחשב X, projection מחשב X — אם זהה, אפשר
   להוריד את ה-DB column בעתיד).
3. **לסמן הצעות פרובידר כ-signed offers** — שדה `signature` ב-`ratson-proposal`,
   מתחיל כ-optional, מתחזק עם הזמן.
4. **לתעד extraction עם הסוכן שביצע אותו** — הוסף שדה `extracted_by_agent`
   ל-Ratson; היום זה תמיד "gemini-via-our-server". בעתיד יהיה "user-chose-agent-X".
5. **לפצל את `concierge-extract` ל-2 endpoints**: `/extract` (טהור AI,
   חתום בעתיד) ו-`/enrich` (אינדקס, אבחנה זמנית). הפיצול הזה עצמו הוא
   הכנה ל-§3.1 ו-§3.2.

ארבעת הצעדים האלה מקדמים את הקונסיירז' למסלול ה-serverless בלי לדרוש שום
שינוי UX, ובלי לדחות את ההשקה.

## 7. שתי הצהרות סיכום

1. **הצד הקבוצתי של הקונסיירז' הוא serverless-ready ברגע ש-Phase 0
   מוטמע**. אין שם בעיה ארכיטקטונית פתוחה — רק עבודה. ה-`evaluateRatsonConsensus`
   הופך לפונקציה טהורה; ה-`willingnessModel` הופך לחישוב צפוי שאפשר
   לאמת מבחוץ.

2. **הצד הציבורי של הקונסיירז' (חילוץ, התאמה, פרובידרים) הוא הבעיה
   הקשה ביותר בכל הארכיטקטורה — אבל גם המקום שבו הפלורליזם החתום הופך
   ליתרון תחרותי**. אורקל יחיד = פגיע. רשת אורקלים חתומה = יציב, אמין,
   שקוף — וקשה לחיקוי. הקונסיירז' הוא לא רק "פיצ'ר לבזר" — הוא הזירה
   שבה הביזור מקבל את המשמעות הכלכלית שלו.
